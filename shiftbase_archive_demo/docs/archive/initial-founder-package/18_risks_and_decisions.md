# 리스크와 의사결정

## 핵심 리스크 1: 영상 없는 강의 플랫폼으로 오해받을 수 있음

### 설명

사용자가 기존 강의 플랫폼에 익숙하다면 영상을 기대할 수 있습니다. 영상이 없으면 콘텐츠가 부족하다고 느낄 수 있습니다.

### 대응

- 처음부터 영상 강의가 아니라 실습 인터페이스라고 명확히 설명
- 실습 화면 미리보기 제공
- 결과물과 템플릿을 전면에 배치
- 단순 수강이 아니라 업무흐름 전환이라는 표현 사용

## 핵심 리스크 2: 실습 인터페이스 개발 비용이 커질 수 있음

### 설명

모든 모듈을 개별 UI로 만들면 개발 부담이 커집니다.

### 대응

- 재사용 가능한 실습 엔진 구축
- Inquiry, Classification, Draft, Checklist, Report 컴포넌트 표준화
- 초기에는 고객문의 모듈 하나에 집중
- 시나리오 데이터 기반 렌더링 도입

## 핵심 리스크 3: AI 결과 품질 불안정

### 설명

AI가 잘못된 답변을 생성하면 학습 품질과 신뢰에 문제가 생깁니다.

### 대응

- AI 없이도 진행 가능한 예시 결과 제공
- AI 결과는 초안으로 표시
- 체크리스트 검토를 필수화
- 위험 업종과 민감정보 안내
- 사용자 수정과 검토를 학습 과정에 포함

## 핵심 리스크 4: 소규모 업체가 돈을 내지 않을 수 있음

### 설명

소규모 업체는 교육 비용 지불 의향이 낮을 수 있습니다.

### 대응

- 저가 단일 모듈과 템플릿 팩 제공
- 무료 체험으로 가치 확인
- B2B와 기관 교육으로 확장
- 적용 사례와 시간 절감 지표 확보

## 핵심 리스크 5: 너무 넓은 카테고리로 흐려질 수 있음

### 설명

소상공인, 제조업, 보안, 바이브코딩, 창업문서까지 모두 하려 하면 브랜드가 흐려질 수 있습니다.

### 대응

- 초기에는 소규모 업체 AX에 집중
- 공통 메시지는 내 일에 AI를 더하다로 유지
- 카테고리는 단계적으로 확장
- 모든 모듈은 업무흐름 전환이라는 공통 구조를 따름

## 핵심 리스크 6: 법적 책임과 개인정보 이슈

### 설명

고객응대, 의료, 금융, 교육 상담 등에서 부정확한 AI 안내가 문제가 될 수 있습니다.

### 대응

- 실제 개인정보 입력 경고
- AI 결과 검토 고지
- 업종별 제한 문구
- 민감 상담은 일반 운영 안내로 제한
- 이용약관과 개인정보 처리방침 준비

## 핵심 의사결정

### 결정 1: 초기 영상 없음

결정:

- 초기 제품에는 영상 강의를 넣지 않음

이유:

- 차별점은 웹 실습 인터페이스
- 영상 제작 비용과 시간이 큼
- 영상이 들어가면 기존 LMS로 보일 수 있음

### 결정 2: 첫 모듈은 고객문의 자동응답

결정:

- 첫 MVP는 고객문의 자동응답과 상담 분류

이유:

- 소규모 업체가 쉽게 이해
- AI 적용 지점이 명확
- 실습 인터페이스 구현 가능
- 결과물이 명확

### 결정 3: 초기 스택은 Next.js와 Supabase

결정:

- Next.js, TypeScript, Tailwind, shadcn/ui, Supabase로 시작

이유:

- 빠른 UI 구현
- DB와 인증 통합
- 실습 상태 저장 가능
- 1인 개발에 현실적

### 결정 4: AI는 보조 기능으로 시작

결정:

- 실습 엔진을 먼저 만들고 AI 기능은 점진적으로 추가

이유:

- 비용과 품질 리스크 감소
- 학습 경험이 AI 호출에 종속되지 않음

## 보류 중인 질문

- 브랜드명 Shiftbase의 상표와 도메인 사용 가능성
- 첫 유료 가격대
- B2C 우선인지 B2B 파일럿 우선인지
- 실습 결과 저장을 회원가입 전에도 허용할지
- 실제 고객 데이터 업로드를 언제 허용할지
- 모바일 실습을 어느 수준까지 지원할지

## 피벗 가능성

### 피벗 1: 소규모 업체 특화

만약 소규모 업체 반응이 좋으면, 카테고리를 깊게 확장합니다.

### 피벗 2: 기업교육 특화

만약 B2C 결제가 약하고 기업 문의가 강하면, 팀 교육과 리포트 중심으로 확장합니다.

### 피벗 3: 템플릿과 플레이북 판매

실습 제품 개발이 무거우면, 템플릿 팩과 플레이북을 먼저 유료화합니다.

### 피벗 4: 내부 도구 제작 교육

바이브코딩과 업무 도구 제작 수요가 강하면, 실습형 개발 인터페이스로 확장합니다.

## 핵심 판단

가장 큰 위험은 제품을 너무 크게 시작하는 것입니다. 단일 실습 모듈의 완성도가 초기 성공의 핵심입니다.

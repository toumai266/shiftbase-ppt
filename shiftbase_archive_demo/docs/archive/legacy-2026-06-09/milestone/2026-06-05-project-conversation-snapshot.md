## 1. 이 문서의 목적

이 문서는 지금까지의 대화, 현재 코드 상태, 기존 전략 문서, 콘텐츠 가치 평가 문서를 종합해 Shiftbase 프로젝트의 현재 위치를 기록한다.

목적은 다음과 같다.

- 프로젝트가 무엇인지 다시 말로 붙잡는다.
- 지금 무엇이 만들어져 있고 무엇이 아직 초안인지 구분한다.
- 사용자가 왜 이 프로젝트를 계속 붙잡고 있는지 가치 관점에서 정리한다.
- "지금 당장 뭘 해야 한다"가 아니라, 어떤 판단이 필요한지 대화의 재료를 남긴다.

이 문서는 로드맵이나 PRD가 아니다. 지금 단계에서는 해결책보다 사고의 선명도가 더 중요하다.

## 2. 프로젝트 개요

Shiftbase는 단순한 AI 강의 사이트가 아니다.

현재까지의 구조를 종합하면, Shiftbase는 다음 세 가지가 겹친 프로젝트다.

1. AI 업무 전환용 인터랙티브 웹북
2. 컨테이너 단위의 로컬 CMS/저작도구
3. 컨테이너, 모듈, 페이지, 블록을 평가하는 멀티 모델 평가 토론장

기존 용어 문서에서는 Shiftbase를 "도서형 설명글과 웹 기반 인터랙션을 결합한 페이지형 학습 플랫폼"으로 정의한다. 사용자는 긴 글을 일방적으로 읽는 대신, 좌측의 설명을 읽고 우측의 워크스페이스에서 직접 보고, 조작하고, 판단한다.

핵심 단위는 `Container`다. 컨테이너는 일반적인 강의 묶음이 아니라, 하나의 학습 목표와 산출물을 가진 짧은 인터랙티브 학습서다.

현재 제품의 더 실제적인 표현은 다음에 가깝다.

> AI를 "쓰는 법"보다, AI가 만든 초안과 자동화 결과를 사람이 어떻게 판단하고 고칠지 배우는 실무형 웹북.

이 표현은 기존 사업전략 문서의 핵심 문장과도 맞닿아 있다.

> AI 초안을 사람이 판단하는 실무 웹북.

## 3. 지금까지 대화에서 드러난 사용자 의도

최근 대화에서 사용자는 기능 추가보다 프로젝트의 품질 기준과 운영 방식을 계속 재정의하고 있다.

핵심 발화는 다음 흐름으로 요약된다.

### 3-1. 평가 에이전트 토론장

사용자는 각 컨테이너, 모듈, 페이지를 평가하는 "평가 에이전트 토론장"을 원한다.

형태는 ChatGPT 같은 채팅 인터페이스다. 사용자가 총괄로서 특정 이슈를 던지면 GPT, Claude, Gemini, NVIDIA, OpenRouter 기반 모델 등 여러 모델이 각자의 관점과 페르소나로 평가하고 개선안을 제출한다.

중요한 점은 이 기능이 단순 자동 점수표가 아니라는 점이다. 사용자는 "여러 모델, 여러 관점, 여러 페르소나"를 통해 콘텐츠의 약점을 드러내고 싶어 한다.

### 3-2. 가짜 데이터와 데모성 폴백에 대한 강한 거부

사용자는 과도한 폴백, 데모, 가짜 데이터가 남는 것을 강하게 거부했다.

이 발화는 단순 취향이 아니라 프로젝트의 신뢰 기준이다. Shiftbase는 "AI가 만든 콘텐츠를 사람이 판단한다"는 메시지를 팔려고 한다. 그러므로 제품 내부가 더미 데이터, 가짜 성공, 과한 fallback으로 돌아가면 메시지 자체가 무너진다.

현재 저작도구 문서에도 같은 방향이 있다. artifact 생성에서 프록시가 떠 있지 않으면 더미 성공이나 가짜 artifact를 반환하지 않고, 생성 버튼은 비활성 또는 실패 상태가 된다.

### 3-3. 여러 AI 모델 연결

사용자는 에이전트별로 다양한 AI 모델을 사용할 수 있어야 한다고 했다. NVIDIA와 OpenRouter도 명시했다.

현재 코드상 평가 provider는 `openai`, `anthropic`, `gemini`, `openrouter`, `nvidia`를 지원하는 구조가 들어와 있다. 구성은 `SHIFTBASE_EVAL_AGENTS` JSON 배열과 provider별 API 키 환경변수를 통해 이뤄진다.

다만 이것은 "구조가 있다"는 뜻이지, 현재 런타임에서 모든 모델이 연결되어 검증됐다는 뜻은 아니다.

### 3-4. 컨테이너 선택 UX에 대한 문제의식

사용자는 컨테이너 선택을 단순 드롭다운이 아니라 더 사용성 있게 만들라고 했다.

이 요청은 "예쁘게 바꾸라"가 아니라, UX/UI와 유저플로우를 조사하고 반영하라는 요구였다. 평가 토론장에서 컨테이너, 모듈, 페이지, 블록을 선택하는 일이 핵심 행동이 되기 때문에, 대상 선택 UI는 제품의 중심 흐름이다.

현재 `EvaluationArena`에는 검색, 레벨 필터, 현재 선택 표시, 대상 경로 표시가 들어가 있다. 이는 단순 드롭다운보다는 나아진 구조다. 다만 실제 사용자가 많은 컨테이너와 수백 개 페이지 안에서 빠르게 대상을 고를 수 있는지는 별도 사용 검증이 필요하다.

### 3-5. Codex Python SDK 도입 검토

사용자는 Python SDK를 이용해 프로그램 내에서 Codex를 사용할 수 있게 됐다는 소식을 보고, 현재 프로젝트에 도입할 수 있는지 물었다.

확인 결과, 공식 문서 기준 Codex SDK는 존재하며 Python SDK는 로컬 Codex app-server를 JSON-RPC로 제어하는 방식이다. 다만 현재 프로젝트의 컨테이너 내용 생성은 Codex SDK 기반이 아니라 OpenAI OAuth 프록시 뒤의 Responses API 기반으로 구현되어 있다.

현재 생성 경로는 다음에 가깝다.

```text
저작자 자연어 입력
-> /api/local-cms/artifacts/generate 또는 /api/local-cms/reading/generate
-> openai-oauth 프록시
-> /v1/responses
-> markdown 또는 artifact JSON 반환
-> 컨테이너 스펙에 저장
```

Codex SDK는 "컨테이너 본문 생성"보다 "개선안 자동 패치, 파일 수정, 테스트 실행" 같은 작업형 에이전트에 더 잘 맞는 성격으로 보인다. 이 문서에서는 도입 여부를 결론내리지 않는다.

### 3-6. 공개 컨테이너 축소

사용자는 "비전공자 AI"와 "AI비서 만들기" 컨테이너를 제외하고 나머지를 전부 draft 상태로 돌리라고 했다. 이유는 직접 하나씩 수정하면서 런칭하는 편이 낫다고 판단했기 때문이다.

현재 파일 상태 기준 공개 컨테이너는 2개다.

| slug | title | status |
| --- | --- | --- |
| `ai-literacy-starter` | 비전공자를 위한 AI 기본 지식 | published |
| `ai-personal-assistant-after-hours` | 퇴근 후 1시간, 나만의 AI 비서 만들기 | published |

나머지 18개 컨테이너는 `draft`다.

이 결정은 기존 2026-06-01 사업전략 문서의 "메일/문서", "보고서/대시보드" 우선 유료 검증 가설과 다르다. 따라서 현재 프로젝트에는 명시적인 전략 전환이 생겼다.

과거 문서의 우선순위:

- `ai-email-document-communication`
- `ai-report-writing-dashboard`

현재 대화에서 실제 공개 유지한 컨테이너:

- `ai-literacy-starter`
- `ai-personal-assistant-after-hours`

이 차이는 오류라기보다 현재 생각이 이동하고 있다는 증거다.

## 4. 현재 구현 상태

### 4-1. 기술 스택

현재 프로젝트는 Next.js/TypeScript 기반이다.

주요 스크립트는 다음과 같다.

- `npm run dev`
- `npm run build`
- `npm run typecheck`
- `npm run validate:content`
- `npm run test:backend`

콘텐츠 컨테이너는 현재 `content/containers/*/container.json`에 저장되어 있다. `.env.example` 기준 저작 백엔드는 `SHIFTBASE_AUTHORING_BACKEND=local-file`을 기본으로 둔다.

### 4-2. 콘텐츠 계층

기존 문서 기준 콘텐츠 계층은 다음과 같다.

```text
Hub
  Track
    Container
      Module
        Lesson
          Page / Step
            Reading Pane
            Workspace
              Workspace Block
```

중요한 원칙은 "페이지 타입을 늘리는 방식"이 아니라, 좌측 본문과 우측 워크스페이스의 조합으로 학습 경험을 만든다는 점이다.

현재 저작 상태 문서에서는 세부 페이지를 책처럼 좌우 2페이지 구조로 유지한다고 정리한다.

```text
좌측: 학습 본문
우측: 실행형 artifact, 정적 참고 이미지, 매뉴얼 카드, 비교표, 예시, 결과 정리
```

### 4-3. 컨테이너 수와 상태

기존 리뷰 문서들은 2026-06-01 기준 `content/containers` 아래 19개 컨테이너를 분석했다. 현재 파일 기준으로는 20개 컨테이너가 있다. 추가 또는 변경된 컨테이너 흐름을 반영하면, 현재 상태는 다음과 같다.

| 상태 | 개수 |
| --- | ---: |
| published | 2 |
| draft | 18 |
| total | 20 |

현재 published 컨테이너는 입문/신뢰 형성 성격의 `ai-literacy-starter`와 욕망형 제작 경험 성격의 `ai-personal-assistant-after-hours`다.

### 4-4. 컨테이너 내용 생성 기능

컨테이너 내용 생성 기능은 이미 코드상 존재한다.

좌측 리딩 본문 생성:

- `app/api/local-cms/reading/generate/route.ts`
- `components/cms/AuthoringWorkflow.tsx`

우측 artifact 생성:

- `app/api/local-cms/artifacts/generate/route.ts`
- `lib/backend/agentArtifactGenerator.ts`
- `components/cms/AuthoringWorkflow.tsx`

다만 현재 구조는 Codex SDK 기반이 아니라 OpenAI OAuth 프록시와 Responses API 기반이다.

또한 이전 확인에서 `http://127.0.0.1:10531` OAuth 프록시 연결은 실패했다. 따라서 "코드상 생성 흐름이 있다"와 "현재 로컬 런타임에서 바로 생성된다"는 분리해야 한다.

### 4-5. 평가 에이전트 토론장

평가 토론장 골격도 코드상 존재한다.

주요 파일:

- `components/cms/EvaluationArena.tsx`
- `lib/backend/evaluationService.ts`
- `lib/backend/evaluationProviders.ts`
- `lib/backend/evaluationScope.ts`
- `app/api/local-cms/evaluation-threads/*`

현재 구조가 제공하는 것:

- 컨테이너, 모듈, 레슨, 페이지, 블록 단위 평가 scope 생성
- 총괄 이슈 입력
- 평가 스레드 생성
- 참가자 모델 선택
- provider별 실행
- 참가자별 결과 저장
- 종합 요약과 next actions 표시

지원 provider:

- OpenAI
- Anthropic
- Gemini
- OpenRouter
- NVIDIA

중요한 제한:

- `SHIFTBASE_EVAL_AGENTS`에 실제 agent 설정이 필요하다.
- API 키 또는 OAuth 프록시가 없으면 참가자는 비활성 또는 실패 상태가 된다.
- 최대 참가자 수는 run당 4개로 제한되어 있다.
- 평가는 모델 응답을 JSON으로 파싱하는 구조라, provider 응답 품질과 스키마 준수에 영향을 받는다.

## 5. 기존 가치 평가의 핵심

기존 리뷰 문서들은 현재 컨테이너들에 대해 꽤 냉정한 평가를 남겼다.

핵심 결론은 다음과 같다.

> 지금은 "팔 수 있는 목차"에 가깝고, "돈이나 시간을 낼 만한 본문"은 아직 아니다.

또 다른 표현으로는 다음과 같다.

> 주제 포트폴리오와 업무 영역 선정은 좋지만, 실제 콘텐츠 상품 가치와 읽기 매력은 아직 낮다.

이 평가는 단순히 콘텐츠를 깎아내리는 것이 아니다. 오히려 프로젝트의 잠재 가치와 현재 구현 가치를 분리한다.

### 5-1. 잠재 가치는 있는 이유

현재 컨테이너 주제는 한국 실무자가 실제로 반복해서 겪는 문제를 넓게 잡고 있다.

- 메일과 문서 커뮤니케이션
- 회의록과 액션아이템
- 보고서 기획과 초안
- 엑셀 데이터 정리와 분석
- 리서치와 근거 정리
- 고객지원
- HR
- 재무
- 개인정보와 리스크
- 영업 제안
- 총무 운영
- 지식베이스
- AI 비서
- 유튜브 강의 웹북화

주제 포트폴리오 자체는 약하지 않다. 오히려 "AI를 업무에 붙일 때 어디서 막히는가"를 폭넓게 짚고 있다.

### 5-2. 현재 구현 가치가 낮게 평가된 이유

기존 리뷰 문서의 공통 문제의식은 반복성이다.

- 컨테이너마다 문장 구조가 비슷하다.
- 페이지 제목과 체크포인트가 과하게 반복된다.
- "한국 업무 맥락", "승인선", "책임 경계" 같은 표현은 많지만 실제 원문과 전후 비교가 부족하다.
- `code-canvas`, 체크리스트, 예시 목록은 많지만 독자의 자기 업무 자료가 축적되는 느낌은 약하다.
- 인터랙티브 수가 품질을 증명하지 못한다.

기존 리뷰 문서에서는 `676개 페이지`, `228개 code-canvas`, 반복 체크포인트 등의 정적 분석을 근거로 들었다. 이 수치는 "많음"의 증거가 아니라 "반복 위험"의 증거로 쓰였다.

### 5-3. 상용 실용서 수준의 기준

기존 가치 평가 문서는 상용 수준의 최소 조건을 다음 요소들로 정리했다.

1. 실제 원문
2. AI 요청문
3. AI 초안
4. 사람의 판단
5. 최종 산출물
6. 전후 비교
7. 내 업무 적용란
8. 재사용 카드

이 기준은 현재 프로젝트의 핵심 가치 판단에 계속 유효하다.

Shiftbase가 팔려면 "AI를 조심해서 쓰세요"라는 원칙보다, 독자가 자기 업무의 문장과 자료를 바꿔보는 경험이 필요하다.

## 6. 현재 프로젝트의 가치 평가

### 6-1. 제품 가치

가장 큰 제품 가치는 "AI 학습을 읽기와 판단의 경험으로 바꾸려는 시도"에 있다.

시장에는 AI 강의, 프롬프트 모음, 자동화 튜토리얼, 템플릿 묶음이 많다. Shiftbase가 다르게 설 수 있는 지점은 다음이다.

- AI 결과를 그대로 쓰는 것이 아니라 검토하는 법을 가르친다.
- 단순 도구 사용법보다 업무 장면과 책임 경계를 다룬다.
- 좌측 본문과 우측 작업면을 결합해 책과 도구 사이의 경험을 만든다.
- 컨테이너 단위로 산출물을 남기는 구조를 지향한다.

이 가치가 실제로 전달되려면 콘텐츠가 더 구체적이어야 한다. 그러나 방향 자체는 흔한 AI 교육과 다르다.

### 6-2. 사용자 가치

예상 사용자는 "AI를 아예 모르는 사람"만이 아니다. 오히려 다음 상태의 사용자가 더 중요할 수 있다.

- AI를 써봤지만 결과를 믿기 어렵다.
- AI 답변을 상사, 고객, 팀원에게 공유하기 불안하다.
- 프롬프트는 따라 썼지만 자기 업무에 적용이 잘 안 된다.
- 자동화 전에 무엇을 맡기고 무엇을 사람이 승인해야 할지 모르겠다.

현재 공개 유지된 두 컨테이너는 이 사용자 여정의 앞부분을 맡는다.

`비전공자를 위한 AI 기본 지식`은 신뢰 형성과 진입 장벽 완화 역할을 한다. 다만 기존 리뷰에서는 범용성이 강해 단독 상품 매력은 낮다고 평가됐다. 이 컨테이너는 "돈을 버는 주력 상품"보다 "오해를 줄이는 입구"로 보는 편이 더 자연스럽다.

`퇴근 후 1시간, 나만의 AI 비서 만들기`는 욕망형 진입점이다. 사용자가 "내가 실제로 뭔가 만들 수 있다"는 감각을 얻기 쉽다. 동시에 개인정보, 권한, 승인, 실패 시 멈춤 같은 프로젝트의 핵심 가치도 넣을 수 있다.

### 6-3. 사업 가치

사업적으로는 아직 검증 전이다.

기존 전략 문서는 메일/문서와 보고서/대시보드를 1차 유료 검증 후보로 봤다. 이유는 지불 의향이 명확하고, 실무자의 반복 고통과 가깝기 때문이다.

하지만 현재 대화에서는 공개 범위를 비전공자 AI와 AI비서 만들기로 좁혔다.

이 변화는 다음 질문을 만든다.

- 지금 프로젝트는 당장 유료 검증보다 신뢰 형성과 첫 경험을 먼저 만들려는가?
- 유료 가치가 큰 업무형 컨테이너를 잠시 숨기고, 입문/욕망형 컨테이너로 제품 감각을 다듬으려는가?
- 아니면 기존 유료 전략 자체를 다시 생각해야 하는가?

이 문서는 답을 정하지 않는다. 다만 현재 상태에서는 "무엇을 공개할 것인가"와 "무엇으로 돈을 받을 것인가"가 같은 질문이 아닐 수 있다.

### 6-4. 기술 가치

기술적으로는 다음 자산이 이미 있다.

- 파일 기반 컨테이너 스펙
- 로컬 CMS
- 좌우 저작 화면
- AI 본문 생성
- AI artifact 생성
- sandbox iframe 기반 artifact 렌더링
- 평가 스레드/토론장
- 여러 provider 지원 구조
- 컨테이너/모듈/레슨/페이지/블록 단위 평가 scope

이는 작은 프로젝트치고 적지 않은 기반이다.

하지만 기술 가치의 핵심은 "기능이 많다"가 아니라, 콘텐츠 품질을 올리는 루프를 만들 수 있다는 점이다.

```text
컨테이너 작성
-> 평가 이슈 제기
-> 여러 모델/페르소나 평가
-> 총괄 판단
-> 직접 수정
-> 다시 평가
```

이 루프가 잘 작동하면 Shiftbase는 단순 CMS보다 "콘텐츠 품질 관리 시스템"에 가까워진다.

### 6-5. 브랜드 가치

브랜드 관점에서 가장 조심해야 할 것은 "AI로 대량 생성한 웹북"처럼 보이는 것이다.

기존 가치 평가 문서의 가장 큰 경고도 이 지점이다. 주제가 아무리 좋아도 문장, 예시, 인터랙티브가 반복되면 사용자는 금방 알아차린다.

따라서 브랜드 가치는 다음 약속 위에 있어야 한다.

- 빠른 AI 생성보다 사람의 판단을 우선한다.
- 많은 컨테이너보다 하나의 깊은 장면을 우선한다.
- 더미 성공보다 실패를 솔직히 드러낸다.
- 범용 원칙보다 실제 업무 원문과 전후 비교를 보여준다.

## 7. 현재의 긴장과 모순

프로젝트가 나쁘다는 뜻이 아니라, 지금 생각해야 할 긴장이 분명히 있다.

### 7-1. 입문 컨테이너를 공개할 것인가, 유료 가치 컨테이너를 공개할 것인가

현재 published로 남긴 두 컨테이너는 입문성과 욕망성이 강하다.

하지만 기존 가치 평가에서 상업적 우선순위가 높게 나온 것은 메일, 회의록, 보고서, 엑셀, 리서치, 리스크 쪽이다.

이 차이를 없애려고 성급히 하나를 버릴 필요는 없다. 오히려 다음 질문을 남길 수 있다.

- 공개 컨테이너는 신뢰 형성용인가?
- 유료 컨테이너는 별도 검증용인가?
- 사용자는 먼저 "AI를 이해"해야 하는가, 아니면 먼저 "내 업무 문제"를 봐야 하는가?

### 7-2. 자동 생성 기능과 수작업 품질 관리

현재 프로젝트에는 AI로 본문과 artifact를 생성하는 기능이 있다.

동시에 사용자는 "직접 하나씩 수정하면서 런칭하는 게 낫겠다"고 판단했다.

이 둘은 충돌처럼 보이지만, 꼭 그렇지는 않다. 생성 기능은 초안을 빠르게 만들 수 있다. 그러나 현재 프로젝트의 품질 기준은 초안 생성이 아니라 사람이 고친 최종본이다.

즉, 생성 기능은 생산 도구일 수 있지만, 런칭 기준은 아니다.

### 7-3. 평가 에이전트와 총괄자의 권한

평가 토론장은 여러 모델의 판단을 모은다.

하지만 최종 판단자는 모델이 아니라 총괄자다. 이 구조가 중요하다.

Shiftbase가 말하는 "AI 초안을 사람이 판단한다"는 원칙은 제품 제작 과정에도 그대로 적용된다. 평가 에이전트는 권위자가 아니라 반대 의견과 개선안을 내는 참가자다.

### 7-4. 기능 완성도와 콘텐츠 완성도

기술적으로는 많은 것이 들어왔다.

하지만 사용자가 지금 고민하는 핵심은 기능 부족보다 콘텐츠와 상품의 기준선에 가깝다.

이 프로젝트가 실패할 수 있는 길은 "기능이 없어서"보다 "기능은 많은데 읽고 싶은 컨테이너가 없어서"일 가능성이 더 크다.

## 9. 프로젝트를 보는 현재 관점

지금 프로젝트는 "완성 직전"이라기보다 "정체성을 선명하게 만들어야 하는 단계"에 가깝다.

좋은 점:

- 문제 영역은 좋다.
- 기술 기반은 이미 적지 않다.
- 평가 토론장이라는 품질 관리 장치가 프로젝트 철학과 잘 맞는다.
- 사용자가 양보다 깊이를 선택하기 시작했다.

위험한 점:

- 콘텐츠가 대량 생성 초안처럼 보일 위험이 있다.
- 기존 문서와 현재 판단 사이에 우선순위 차이가 생겼다.
- 기능 개발이 콘텐츠 품질 문제를 가릴 수 있다.
- AI 생성 기능이 품질 기준으로 오해될 수 있다.
- "AI 교육"과 "AI 업무 판단 웹북" 사이의 표현이 흔들릴 수 있다.

가장 중요한 변화:

> 전체를 공개해서 커 보이게 만드는 방향에서, 일부만 남기고 직접 고쳐 런칭하는 방향으로 이동했다.

이 변화는 프로젝트에 좋은 신호일 수 있다. 현재 가장 큰 리스크가 수량 부족이 아니라 밀도 부족이기 때문이다.

## 10. 계속 이야기해볼 질문

이 문서는 해결책을 제안하기보다 다음 대화를 위한 질문을 남긴다.

### 10-1. 제품 정체성

- Shiftbase는 "AI 강의 플랫폼"인가, "AI 업무 판단 웹북"인가, "콘텐츠 품질 평가 CMS"인가?
- 사용자가 첫 화면에서 느껴야 하는 것은 "배울 수 있다"인가, "내 일을 바꿀 수 있다"인가?
- 웹북이라는 단어는 사용자에게 충분히 이해되는가?

### 10-2. 공개 컨테이너의 역할

- `비전공자를 위한 AI 기본 지식`은 독립 컨테이너인가, 입문 관문인가?
- `퇴근 후 1시간, 나만의 AI 비서 만들기`는 욕망형 무료/저가 유입인가, 핵심 상품인가?
- 기존 유료 후보였던 메일/문서와 보고서는 왜 지금 숨기는 것이 맞다고 느껴졌는가?

### 10-3. 콘텐츠 품질

- 사용자가 "이건 내 얘기다"라고 느끼는 첫 장면은 어디에 있어야 하는가?
- 현재 공개 컨테이너 2개는 실제 원문, 실패 예, 전후 비교, 사람 판단을 충분히 갖고 있는가?
- 한 컨테이너를 런칭 가능하다고 부르는 기준은 무엇인가?

### 10-4. 평가 토론장

- 평가 에이전트는 점수표인가, 비평가인가, 편집자 회의인가?
- 모델별 페르소나는 어떻게 달라야 실제로 다른 관점을 주는가?
- 총괄자는 평가 결과를 어떻게 읽고, 어떤 순간에 반려해야 하는가?

### 10-5. 사업 판단

- 지금 필요한 것은 유료 검증인가, 신뢰 형성인가, 제품 감각 확인인가?
- 첫 유료 결제는 어떤 감정에서 나와야 하는가: 불안 감소, 시간 절약, 결과물 완성, 자기계발 욕망?
- 개인 사용자가 먼저인가, 팀/조직 파일럿이 먼저인가?

## 12. 마일스톤 메모

2026-06-05 현재 이 프로젝트에서 가장 중요한 감각은 "많이 만들었다"가 아니다.

더 중요한 감각은 다음이다.

> 이 프로젝트는 AI가 만든 것을 사람이 판단하는 제품을 만들면서, 동시에 자기 자신도 AI가 만든 초안을 사람이 판단해 고치는 방식으로 완성되어야 한다.

따라서 현재의 draft 전환은 후퇴라기보다 품질 기준을 다시 세우는 행위에 가깝다.

지금은 해결책을 확정하기보다, 무엇을 공개할지, 무엇을 숨길지, 무엇을 직접 고쳐야 할지, 어떤 경험을 사용자가 처음 느껴야 할지를 계속 말로 정리해야 하는 시점이다.

# Shiftbase 프로젝트 현재 상태와 생각 정리

작성일: 2026-06-05  
목적: 해결책을 정하기보다, 지금까지의 대화와 기존 문서를 바탕으로 프로젝트의 현재 형태, 가치, 긴장, 판단 재료를 한곳에 묶는다.  
성격: 현재 사고의 milestone. 실행 계획서나 PRD가 아니다.

## 1. 프로젝트 개요

Shiftbase는 단순한 AI 강의 사이트가 아니다.

현재 문서 기준의 정의는 "도서형 설명글과 웹 기반 인터랙션을 결합한 페이지형 학습 플랫폼"이다. 사용자는 긴 글을 스크롤로 읽는 대신 한 화면씩 넘기며, 좌측에서는 본문을 읽고 우측에서는 시각화, 매뉴얼, 비교표, 입력 도구, 체크리스트, 산출물 카드, 실행형 artifact 등을 다룬다.

기존 문서가 정리한 핵심 경험은 다음과 같다.

```text
읽기 -> 보기 -> 조작하기 -> 판단하기 -> 적용하기 -> 산출물 가져가기
```

이 흐름에서 중요한 것은 "AI를 쓰는 방법" 자체가 아니다. 프로젝트가 점점 선명해진 방향은 "AI가 만든 결과를 사람이 어떻게 판단하고, 자기 업무에서 쓸 수 있는 형태로 고치는가"에 가깝다.

따라서 Shiftbase의 가치는 다음 세 가지가 겹치는 지점에 있다.

- 웹북: 책처럼 읽히되, 페이지마다 화면 단위 경험이 있다.
- 실무 판단 훈련: 프롬프트 복사보다 결과 검토, 수정, 승인, 공유 기준을 배운다.
- 저작/평가 시스템: 컨테이너, 모듈, 페이지, artifact를 만들고 평가하는 내부 도구가 있다.

## 2. 현재 대화에서 드러난 방향

최근 대화에서 사용자는 해결책보다 생각의 선명화를 원한다고 밝혔다. 이 말은 중요하다. 지금 필요한 것은 기능을 더 추가하는 속도가 아니라, 프로젝트가 무엇으로 읽혀야 하는지와 어떤 품질 기준을 가져야 하는지 확인하는 대화다.

대화에서 나온 주요 요구와 판단은 다음과 같다.

- 각 컨테이너, 모듈, 페이지를 평가하는 "평가 에이전트 토론장"을 만들고 싶다.
- 총괄자인 사용자가 특정 이슈를 던지면 GPT, Claude, Gemini, OpenRouter, NVIDIA 등 여러 모델과 여러 페르소나가 평가하고 개선안을 제출하는 구조를 원한다.
- 과도한 폴백, 데모, 가짜 데이터는 제거해야 한다. 신뢰를 팔아야 하는 프로젝트에서 가짜 성공은 제품 정체성을 망친다.
- 에이전트별로 다양한 AI 모델을 선택할 수 있어야 한다.
- 컨테이너 선택 UI가 단순 드롭다운처럼 느껴지면 안 된다. 사용자가 어떤 컨테이너를 고르는지, 어떤 흐름으로 들어가는지, 어디서 비교하고 선택하는지가 중요하다.
- Codex Python SDK는 최근 확인된 가능성이지만, 현재 컨테이너 내용 생성 흐름과 동일한 것은 아니다. 현재 생성 흐름은 OAuth 프록시 뒤의 Responses API 호출이고, Codex SDK는 로컬 Codex app-server를 통한 작업형 에이전트 제어에 가깝다.
- 기존 컨테이너 대부분은 직접 하나씩 수정하면서 런칭하는 편이 낫다고 판단했다.

이 대화의 핵심은 "더 많은 콘텐츠를 노출하자"가 아니라 "보여줄 수 있는 것만 남기고, 나머지는 다시 판단하자"에 가깝다.

## 4. 현재 제품 기능 상태

현재 프로젝트는 Next.js 기반 웹 애플리케이션이다. `package.json` 기준 주요 스크립트는 `dev`, `build`, `typecheck`, `validate:content`, `test:backend`다.

저작도구 쪽은 이미 상당한 구조를 갖고 있다.

- 컨테이너 스펙을 로컬 파일 또는 SQLite 기반으로 읽고 쓸 수 있다.
- 좌측 본문은 `bodyMarkdown`을 기준으로 편집한다.
- 우측 workspace는 정적 참고 자료, 매뉴얼, 비교표, 체크리스트, 결과 카드, `code-canvas` 기반 artifact를 담을 수 있다.
- `code-canvas` artifact는 sandbox iframe에서 렌더링된다.
- AI artifact 생성은 `/api/local-cms/artifacts/generate`를 통해 동작하도록 설계되어 있다.
- 좌측 본문 생성은 `/api/local-cms/reading/generate`를 통해 동작하도록 설계되어 있다.

다만 생성 기능은 "코드상 구현"과 "현재 런타임이 준비됨"을 나눠 봐야 한다. 기존 확인 기준으로는 OAuth 프록시가 켜져 있어야 생성 버튼이 실제로 의미를 갖는다. 더미 성공이나 가짜 artifact를 반환하지 않는다는 원칙은 제품 신뢰 관점에서 중요하다.

평가 에이전트 쪽도 코드상 기초가 있다.

- `.env.example`에는 `SHIFTBASE_EVAL_AGENTS`가 있고, agent별 `id`, `label`, `provider`, `model`, `persona`, `rubric`를 요구한다.
- 지원 provider는 `openai`, `anthropic`, `gemini`, `openrouter`, `nvidia`로 정의되어 있다.
- `lib/backend/evaluationProviders.ts`에는 provider별 호출 함수와 참가자 catalog 로딩, API key 확인, JSON 평가 응답 스키마가 있다.
- 평가 대상은 컨테이너, 모듈, 레슨, 페이지, 블록 단위로 좁힐 수 있는 구조가 있다.

즉, "평가 에이전트 토론장"은 완전히 상상 속 기능은 아니다. 이미 프로젝트 안에 뼈대가 있다. 다만 이것이 제품 경험으로 충분히 다듬어졌는지는 별개의 문제다.

## 5. 기존 문서가 말하는 가치 평가

`docs/reviews/ai-webbook-container-value-evaluation-2026-06-01.md`의 핵심 평가는 강하다.

현재 구현물은 주제 포트폴리오로는 가치가 있지만, 현재 콘텐츠 자체의 매력과 상용 readiness는 낮다는 판단이다. 문서의 표현을 요약하면, 지금은 "팔 수 있는 목차"에 가깝고 "돈이나 시간을 낼 만한 본문"은 아직 아니라는 쪽이다.

이 평가는 프로젝트를 부정하는 것이 아니라, 가치를 분리해서 보자는 의미로 읽힌다.

- 주제 잠재 가치: 높음
- 현재 구현 가치: 보통 이하
- 읽고 싶은 매력: 낮음
- 상용 실용서 readiness: 낮음
- 리팩터링 우선도: 높음

특히 반복 문장, 동일 체크포인트, 비슷한 인터랙티브 구성, 실제 업무 원문 부족, 산출물 샘플 부족이 문제로 지적되어 있다. 이는 사용자가 말한 "드래프트로 내리고 직접 수정하며 런칭"하겠다는 판단과 잘 맞는다.

## 6. 전략 문서가 말하는 사업 가설

`docs/strategy/ai-webbook-business-strategy-2026-06-01.md`와 `docs/reviews/ai-webbook-strategy-feedback-synthesis-2026-06-02.md`는 전략을 다음 방향으로 좁힌다.

Shiftbase는 "AI 자동화"나 "프롬프트 모음"으로 팔면 약하다. 더 선명한 기회는 실제 업무 장면에서 AI 초안을 사람이 판단하고 고치는 경험에 있다.

기존 전략 문서의 한 문장 전략은 다음과 같다.

> Shiftbase AI 웹북은 메일, 회의록, 보고서, 엑셀 같은 실제 업무 장면에서 AI 결과를 어디까지 믿고 어디서 멈춰야 하는지 배우는 실습형 웹북으로 포지셔닝한다.

전략 문서가 원래 6월 유료 검증 후보로 좁힌 것은 `ai-email-document-communication`과 `ai-report-writing-dashboard`였다. 하지만 현재 대화에서는 일단 `ai-literacy-starter`와 `ai-personal-assistant-after-hours`만 published로 남겨졌다. 이 둘은 사업전략 문서의 유료 검증 후보와 정확히 같지는 않다.

이 차이는 모순이라기보다 현재 생각이 움직이고 있다는 신호로 볼 수 있다.

- 전략 문서는 유료 검증 관점에서 메일/문서와 보고서/대시보드를 강하게 본다.
- 현재 공개 상태는 비전공자 입문과 AI 비서를 남겨, 더 넓은 진입과 욕망형 유입을 우선 남겼다.
- 사용자는 지금 당장 해결책보다 프로젝트의 생각을 명확히 하고 싶다고 했다.

따라서 현 시점의 중요한 질문은 "어느 쪽이 맞는가"가 아니라, 공개 상태와 유료 검증 전략이 서로 어떤 역할을 가져야 하는가다.

## 7. 가치의 중심

현재까지의 자료와 대화를 종합하면 Shiftbase의 가치는 세 층으로 나뉜다.

첫째, 콘텐츠 주제의 가치다. 한국 실무자가 실제로 겪는 메일, 회의, 보고서, 엑셀, 리서치, 고객지원, 개인정보, HR, 재무, 영업 같은 문제를 폭넓게 잡았다. 이 자체는 가치가 있다.

둘째, 학습 형식의 가치다. 일반 글이나 영상 강의와 달리 좌측 본문과 우측 workspace를 결합해 판단, 조작, 적용, 산출물을 만들 수 있다. 이 형식은 웹북이라는 말이 단순 포장어가 아니라 실제 인터페이스가 될 수 있음을 보여준다.

셋째, 내부 저작/평가 시스템의 가치다. 컨테이너를 스펙화하고, artifact를 생성하고, 여러 모델과 페르소나로 평가할 수 있다면 Shiftbase는 단순 콘텐츠 묶음이 아니라 "콘텐츠 품질을 계속 심사하고 개선하는 시스템"이 된다.

다만 이 세 층이 아직 같은 밀도로 완성된 것은 아니다.

- 콘텐츠 주제는 넓다.
- 학습 형식은 방향이 있다.
- 저작/평가 시스템은 뼈대가 있다.
- 실제 상용 콘텐츠 밀도는 아직 더 엄격한 검토가 필요하다.

## 8. 현재 가장 큰 긴장

이 프로젝트의 긴장은 "플랫폼"과 "작품" 사이에 있다.

플랫폼으로 보면 컨테이너, 모듈, 페이지, artifact, 평가 에이전트, 모델 provider, 저작 CMS가 모두 중요하다. 이 방향은 장기적으로 강하다. 하지만 플랫폼을 먼저 크게 만들면 실제로 팔리는 첫 컨테이너의 밀도가 흐려질 수 있다.

작품으로 보면 하나의 컨테이너가 실제 책처럼 읽히고, 실제 업무 장면처럼 설득되어야 한다. 이 방향은 런칭 품질에는 강하다. 하지만 하나씩 직접 다듬는 방식만으로는 확장성이 느려질 수 있다.

지금 사용자의 판단은 작품 쪽으로 기울어 있다. "직접 하나씩 수정하면서 런칭하는 게 낫겠다"는 말은, 현재 프로젝트가 플랫폼 욕심 때문에 콘텐츠 판단을 희석하면 안 된다는 신호다.

또 다른 긴장은 "AI 생성"과 "가짜 생성" 사이에 있다.

AI로 내용을 만들 수 있다는 것은 강력하지만, 그 결과가 반복 문장, 데모 artifact, 과도한 폴백, 가짜 성공으로 보이면 프로젝트의 핵심 가치와 충돌한다. 이 프로젝트는 AI를 쓰는 프로젝트이지만, 사용자가 가장 강하게 거부한 것도 "AI가 대충 만든 것처럼 보이는 흔적"이다.

## 9. 평가 에이전트 토론장의 의미

평가 에이전트 토론장은 기능이라기보다 제품 철학에 가깝다.

사용자가 총괄로서 이슈를 던지고, 여러 모델과 페르소나가 서로 다른 관점에서 컨테이너, 모듈, 페이지를 평가한다. 이 구조는 단순 자동 평가보다 더 적합해 보인다. 이유는 Shiftbase의 품질 문제가 정답 채점 문제가 아니라 판단의 문제이기 때문이다.

좋은 평가자는 다음을 봐야 한다.

- 이 페이지가 실제 독자에게 필요한 장면에서 시작하는가.
- 본문이 추상 원칙을 반복하는가, 구체 문장을 보여주는가.
- 우측 workspace가 학습 판단을 돕는가, 기능 수만 늘리는가.
- 산출물이 이름만 있는가, 실제로 가져갈 수 있는 형태인가.
- AI 초안과 사람 판단의 차이가 선명한가.
- 한국 실무 맥락이 구호가 아니라 문장과 상황으로 들어가 있는가.

현재 코드상 provider 확장 가능성은 있다. 하지만 토론장의 가치는 provider 수보다 평가 질문의 질에 달려 있다.

## 10. 컨테이너 선택 UX에 대한 현재 인식

사용자는 기존 컨테이너 선택 방식이 드롭다운처럼 느껴지는 것을 문제로 봤다. 이는 단순 UI 취향 문제가 아니다.

컨테이너는 사용자가 실제로 학습을 시작하는 독립 학습 패키지다. 따라서 선택 UI는 단순 목록 선택이 아니라 다음을 도와야 한다.

- 사용자가 지금 어떤 목적의 컨테이너를 보고 있는지 이해한다.
- published와 draft의 차이를 혼동하지 않는다.
- 컨테이너, 모듈, 페이지, 블록 평가 범위를 쉽게 좁힌다.
- 많은 컨테이너가 있을 때도 현재 작업 맥락을 잃지 않는다.
- 사용자가 "내가 지금 무엇을 고르는가"를 눈으로 확인한다.

아직 이 문서는 UI 해결책을 제시하지 않는다. 다만 컨테이너 선택은 단순 form control이 아니라 제품의 정보 구조와 작업 흐름을 드러내는 장면이라는 점을 milestone에 남긴다.

## 11. Codex SDK에 대한 현재 위치

최근 확인한 Codex Python SDK 소식은 프로젝트에 흥미로운 가능성을 준다. 하지만 현재 컨테이너 내용 생성과 혼동하면 안 된다.

현재 프로젝트의 내용 생성 흐름은 OAuth 프록시 뒤의 Responses API 호출에 가깝다. 반면 Codex Python SDK는 로컬 Codex app-server를 제어하며, 작업형 에이전트가 파일을 보고 수정하고 명령을 실행하는 쪽에 더 가깝다.

따라서 Codex SDK는 "본문 생성 버튼을 대체할 기술"이라기보다 다음과 같은 영역과 더 잘 맞을 수 있다.

- 평가 에이전트가 낸 개선안을 실제 파일 패치 후보로 바꾸기
- 컨테이너 스펙을 점검하고 반복 문장이나 구조 문제를 찾아내기
- 수정 후 content validation이나 테스트를 실행하기
- 총괄자가 승인할 수 있는 변경 묶음을 만들기

현재 단계에서는 Codex SDK를 도입할지보다, 어떤 작업을 "모델 응답 생성"으로 둘지와 어떤 작업을 "파일을 만지는 에이전트 작업"으로 둘지 구분하는 것이 더 중요하다.

## 12. 현재 생각의 열린 질문

이 문서는 해결책을 정하지 않기 때문에, 아래 질문은 남겨둔다.

1. 현재 published로 남긴 `ai-literacy-starter`와 `ai-personal-assistant-after-hours`는 브랜드의 첫인상으로 충분한가, 아니면 단지 임시 공개 상태인가?

2. 전략 문서가 강조한 `ai-email-document-communication`과 `ai-report-writing-dashboard`는 유료 검증 후보로 여전히 중심인가, 아니면 현재 생각에서는 뒤로 밀렸는가?

3. Shiftbase의 첫인상은 "비전공자를 위한 친절한 AI 입문"이어야 하는가, "AI 초안을 사람이 고치는 실무 판단 웹북"이어야 하는가, "나만의 AI 비서 만들기" 같은 욕망형 경험이어야 하는가?

4. 평가 에이전트 토론장은 콘텐츠 런칭 전 품질 게이트인가, 저작 과정 중의 동료 리뷰인가, 아니면 사용자가 볼 수 있는 제품 기능인가?

5. 컨테이너 선택 UX는 콘텐츠 탐색 문제인가, CMS 작업 관리 문제인가, 평가 대상 선택 문제인가? 셋을 한 화면에서 다루면 오히려 흐려질 가능성은 없는가?

6. AI 생성 기능은 어느 범위까지 허용되어야 신뢰를 해치지 않는가? 초안 생성, artifact 생성, 개선안 생성, 파일 패치 중 어느 지점에 사람 승인이 반드시 있어야 하는가?

7. "가짜 데이터, 과도한 폴백, 데모 흔적 제거"라는 원칙은 UI, API, 콘텐츠, 평가 결과에서 각각 어떤 기준으로 드러나야 하는가?

8. 이 프로젝트의 첫 유료 가치는 컨테이너 하나의 완성도에서 나오는가, 아니면 컨테이너를 평가하고 개선하는 시스템의 신뢰에서 나오는가?

## 14. 현재 milestone의 결론

지금 Shiftbase는 아이디어가 부족한 상태가 아니다. 오히려 아이디어와 컨테이너 후보가 많아서, 무엇을 공개하고 무엇을 붙잡고 다시 써야 하는지 구분해야 하는 상태다.

현재 가장 설득력 있는 방향은 "AI 교육"이 아니라 "AI 결과를 실무자가 판단하고 공유 가능한 산출물로 바꾸는 웹북"이다. 이 방향은 기존 문서와도 맞고, 평가 에이전트 토론장이라는 새 기능 구상과도 맞다.

다만 현재 공개 상태, 유료 검증 전략, 욕망형 AI 비서 컨테이너, 비전공자 입문 컨테이너가 같은 메시지를 말하고 있는지는 아직 정리되지 않았다.

이 milestone은 그 불일치를 실패로 보지 않는다. 지금 단계의 핵심 재료로 본다. 프로젝트가 더 명확해지려면, 다음 대화에서 바로 해결책으로 뛰어가기보다 "첫 화면에서 사용자가 이 서비스를 무엇으로 이해해야 하는가"를 계속 물어야 한다.

# Interactive Container Authoring Overview

> 최신 작업 기준은 `03_current_authoring_state_2026-05-22.md`를 우선합니다. 이 문서는 초기 블록 조합형 스펙 설계 배경으로 보존합니다.

이 문서는 컨테이너 저작 구조를 코드 하드코딩에서 스펙 기반으로 전환하기 위한 기능 개요입니다.

## 배경

현재 학습 플레이어는 좌측 서술 영역과 우측 워크스페이스를 가진 페이지형 학습 경험을 이미 구현하고 있습니다.

현재 기준 구현:

- `components/lesson/InteractiveLessonPlayer.tsx`
- `lib/lessonContent.ts`

문제는 컨테이너, 모듈, 레슨, 페이지, 워크스페이스 데이터가 코드에 직접 들어가 있다는 점입니다. 프로토타입 단계에서는 빠르지만, 컨테이너 수가 늘어나면 콘텐츠 생산 병목이 됩니다.

## 제품 방향

Shiftbase의 저작 구조는 다음 방향으로 가야 합니다.

```text
콘텐츠 데이터: 코드 밖의 구조화 스펙
인터랙션 로직: 재사용 가능한 워크스페이스 블록 컴포넌트
렌더러: 스펙을 읽어 좌측 서술과 우측 워크스페이스를 생성
```

즉, 콘텐츠 저작자는 컨테이너의 흐름과 블록 조합을 스펙으로 정의하고, 개발자는 고품질 워크스페이스 블록과 렌더러를 제공합니다.

## 핵심 원칙

페이지는 타입을 갖지 않습니다.

대신 모든 페이지는 공통 구조를 가집니다.

```text
Page
  left: 서술 영역
  workspace: 우측 조작 영역
```

워크스페이스 안의 블록이 타입을 가집니다.

```text
Workspace
  blocks:
    - quiz
    - image-viewer
    - block-board
    - checklist
    - result-card
```

이 구조가 중요한 이유는 다음과 같습니다.

- 한 페이지 안에 이미지와 퀴즈를 함께 넣을 수 있습니다.
- 한 페이지 안에 입력 폼과 결과 카드를 함께 넣을 수 있습니다.
- 페이지 유형을 계속 늘리지 않아도 됩니다.
- 콘텐츠 흐름은 자유롭게 만들되, 컴포넌트 품질은 표준화할 수 있습니다.

## PM/PO 관점의 목표

이 기능의 목표는 CMS를 바로 만드는 것이 아닙니다.

초기 목표는 다음입니다.

- 콘텐츠를 코드에서 분리합니다.
- 컨테이너를 스펙 파일로 정의할 수 있게 합니다.
- 좌측 서술과 우측 워크스페이스 조합을 표준화합니다.
- 반복되는 인터랙션을 워크스페이스 블록으로 재사용합니다.
- 현재 하드코딩된 예시 컨테이너를 새 스펙으로 옮길 수 있게 합니다.

## 비목표

초기 버전에서 하지 않을 일:

- 완전한 WYSIWYG CMS
- 외부 강사용 저작 어드민
- 모든 React 컴포넌트를 자유 삽입하는 기능
- 복잡한 권한 관리
- 마켓플레이스형 콘텐츠 등록

초기에는 파일 기반 스펙과 렌더러 분리만으로 충분합니다.

## 기대 효과

- PM/콘텐츠 기획자가 컨테이너 구조를 코드와 분리해 검토할 수 있습니다.
- 같은 워크스페이스 블록을 여러 컨테이너에서 재사용할 수 있습니다.
- 직무별 무료 컨테이너를 빠르게 늘릴 수 있습니다.
- 학습 경험의 품질과 일관성을 유지할 수 있습니다.
- 이후 내부 저작 도구나 CMS로 확장하기 쉬워집니다.

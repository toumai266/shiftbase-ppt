# PRD: Spec-Driven Interactive Container Authoring

## 문서 목적

이 문서는 Shiftbase의 컨테이너 저작 방식을 코드 하드코딩에서 스펙 기반으로 전환하기 위한 제품 요구사항을 정의합니다.

## 문제 정의

현재 컨테이너의 레슨과 페이지 데이터는 코드에 직접 정의되어 있습니다.

이 방식은 초기 프로토타입에는 적합하지만, 다음 단계에서는 병목이 됩니다.

- 콘텐츠 수정에 개발 작업과 배포가 필요합니다.
- PM/콘텐츠 기획자가 컨테이너 흐름을 독립적으로 작성하고 검토하기 어렵습니다.
- 직무별 무료 컨테이너를 빠르게 늘리기 어렵습니다.
- 동일한 인터랙션을 재사용하기 어렵습니다.
- 페이지 구조와 인터랙션 로직이 섞여 장기 유지보수가 어려워집니다.

## 목표

스펙 기반 저작 구조를 도입해 컨테이너를 코드 밖에서 정의하고, 공통 렌더러가 이를 페이지형 학습 경험으로 표시하게 합니다.

목표:

- 컨테이너, 모듈, 레슨, 페이지를 구조화된 스펙으로 정의합니다.
- 모든 페이지는 좌측 서술과 우측 워크스페이스 구조를 유지합니다.
- 우측 워크스페이스는 표준 블록 조합으로 구성합니다.
- 페이지 간 상태를 공유해 누적 학습과 산출물 생성을 지원합니다.
- 현재 하드코딩된 예시 컨테이너를 새 구조로 이전할 수 있게 합니다.

## 비목표

초기 범위에서 제외합니다.

- 완전한 CMS 또는 WYSIWYG 에디터
- 외부 강사 계정과 권한 관리
- 마켓플레이스형 콘텐츠 등록
- 모든 종류의 자유 React 컴포넌트 삽입
- 결제, 수료증, 조직 관리자 기능
- AI 자동 생성 기반 저작 도구

## 주요 사용자

### PM/PO

컨테이너의 제품 구조, 학습 목표, 산출물, 무료/유료 전환 흐름을 설계합니다.

### 콘텐츠 기획자

모듈, 레슨, 페이지 본문, 워크스페이스 블록 조합, 실습 지시문을 작성합니다.

### 개발자

스펙 렌더러와 워크스페이스 블록 컴포넌트를 구현하고, 스펙 검증을 담당합니다.

### 학습자

카드에서 컨테이너를 선택하고, 페이지 단위로 읽고 조작하며, 최종 산출물을 가져갑니다.

## 사용자 스토리

- PM은 컨테이너 카드에 노출될 제목, 대상, 난이도, 소요 시간, 산출물을 스펙에서 정의할 수 있어야 합니다.
- 콘텐츠 기획자는 각 페이지의 좌측 서술과 우측 워크스페이스 블록을 코드 수정 없이 작성할 수 있어야 합니다.
- 개발자는 새 컨테이너를 만들 때 매번 플레이어 코드를 수정하지 않아야 합니다.
- 학습자는 이전 페이지에서 만든 입력값이나 선택값이 이후 페이지에서 다시 사용되는 경험을 해야 합니다.
- 운영자는 어떤 컨테이너가 무료이고 어떤 컨테이너가 유료인지 스펙에서 구분할 수 있어야 합니다.

## MVP 범위

### 1. 파일 기반 컨테이너 스펙

컨테이너를 구조화 파일로 정의합니다.

필수 포함:

- 메타데이터
- 허브와 트랙
- 대상자
- 난이도
- 소요 시간
- 학습 목표
- 산출물
- 모듈
- 레슨
- 페이지
- 워크스페이스 블록

### 2. 스펙 로더

앱은 컨테이너 스펙을 읽어 컨테이너 목록, 상세 정보, 학습 플레이어에 제공해야 합니다.

요구사항:

- slug 기반 조회
- draft/published 상태 구분
- 잘못된 스펙에 대한 명확한 오류 메시지

### 3. 공통 플레이어 렌더러

기존 좌측 서술/우측 워크스페이스 플레이어 구조를 유지하되, 데이터는 스펙에서 읽습니다.

요구사항:

- 페이지 진행
- 목차
- 진행률
- 좌우 영역
- 워크스페이스 블록 렌더링
- 페이지 간 상태 전달

### 4. 워크스페이스 블록 레지스트리

렌더러는 등록된 블록 타입만 표시합니다.

초기 지원 후보:

- `image-viewer`
- `diagram`
- `quiz`
- `block-board`
- `checklist`
- `text-input`
- `prompt-builder`
- `before-after`
- `table`
- `result-card`
- `state-summary`

### 5. 상태 모델

학습자가 만든 결과를 페이지 간 공유할 수 있어야 합니다.

예시:

- 업무 블럭 배치 결과
- 선택한 AI 적용 후보
- 작성한 프롬프트
- 체크리스트 선택값

### 6. 기존 컨테이너 마이그레이션

현재 하드코딩된 `내 업무를 AI로 전환하기` 예시를 스펙 기반 구조로 이전할 수 있어야 합니다.

이전 성공 기준:

- 기존 좌측 설명 내용이 스펙으로 이동합니다.
- 기존 타임라인 빌더와 분석 워크스페이스가 블록으로 표현됩니다.
- 학습 진행 흐름이 유지됩니다.

## 요구사항

### Functional Requirements

- 컨테이너 목록은 스펙 메타데이터에서 생성되어야 합니다.
- 컨테이너 상세 페이지는 스펙의 outcomes, outputs, modules를 표시할 수 있어야 합니다.
- 학습 플레이어는 스펙의 pages를 순서대로 렌더링해야 합니다.
- 각 페이지는 left와 workspace를 반드시 가져야 합니다.
- workspace.blocks는 여러 개를 가질 수 있어야 합니다.
- 각 블록은 type에 따라 등록된 컴포넌트로 렌더링되어야 합니다.
- 블록은 state를 읽고 쓸 수 있어야 합니다.
- 잘못된 블록 타입이나 누락 필드는 빌드 또는 개발 환경에서 빠르게 발견되어야 합니다.

### Non-Functional Requirements

- 스펙은 사람이 읽고 리뷰하기 쉬워야 합니다.
- 동일한 컨테이너는 코드 변경 없이 텍스트와 구성을 수정할 수 있어야 합니다.
- 모바일에서도 좌측/우측 구조가 자연스럽게 단일 흐름으로 전환되어야 합니다.
- 워크스페이스 블록은 재사용 가능해야 합니다.
- 스펙이 늘어나도 플레이어 코드가 컨테이너별로 분기하지 않아야 합니다.

## Acceptance Criteria

- 신규 컨테이너 1개를 코드 하드코딩 없이 스펙 파일로 추가할 수 있습니다.
- 컨테이너 카드가 스펙 메타데이터에서 생성됩니다.
- 학습 플레이어가 스펙의 모듈, 레슨, 페이지를 표시합니다.
- 한 페이지의 워크스페이스에 2개 이상의 블록을 조합할 수 있습니다.
- 이전 페이지에서 생성한 상태를 이후 페이지의 블록이 읽을 수 있습니다.
- 등록되지 않은 블록 타입을 사용하면 명확한 오류가 발생합니다.
- 기존 예시 컨테이너를 새 스펙 구조로 옮기는 데 성공합니다.

## Release Plan

### Phase 1: 스펙 구조 도입

- 컨테이너 스펙 타입 정의
- 파일 기반 샘플 스펙 작성
- 스펙 로더 작성
- 기본 검증 규칙 추가

### Phase 2: 플레이어 연결

- 기존 플레이어가 스펙 데이터를 사용하도록 연결
- 좌측 서술 렌더러 일반화
- 워크스페이스 블록 레지스트리 도입

### Phase 3: 기존 컨테이너 이전

- 현재 `lib/lessonContent.ts` 기반 콘텐츠를 스펙으로 이전
- 타임라인 빌더를 블록 모델로 표현
- 상태 공유 흐름 점검

### Phase 4: 저작 생산성 개선

- 샘플 컨테이너 템플릿 추가
- 스펙 작성 가이드 추가
- 로컬 프리뷰 또는 개발용 검증 명령 검토

## Risks

- 스펙 자유도가 너무 높으면 렌더러와 모바일 대응이 복잡해질 수 있습니다.
- 스펙 자유도가 너무 낮으면 콘텐츠 기획자가 원하는 페이지 구성을 만들기 어렵습니다.
- 상태 모델을 늦게 설계하면 산출물 생성 흐름이 다시 하드코딩될 수 있습니다.
- CMS를 너무 빨리 만들면 핵심 학습 포맷보다 어드민 구현에 리소스가 쏠릴 수 있습니다.

## Open Questions

- 스펙 파일 형식은 YAML, JSON, MDX 중 무엇을 우선할 것인가?
- 콘텐츠 본문에 Markdown을 어느 범위까지 허용할 것인가?
- 컨테이너 스펙을 앱 번들에 포함할 것인가, 별도 콘텐츠 저장소로 분리할 것인가?
- 학습자 상태 저장은 로컬 우선인가, 계정 기반 저장인가?
- 무료/유료 접근 제어는 스펙 메타데이터만으로 충분한가?
- 직무별 무료 컨테이너 시리즈를 어떤 순서로 이전/제작할 것인가?

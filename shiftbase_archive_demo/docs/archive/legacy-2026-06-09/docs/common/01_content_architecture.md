# Content Architecture

이 문서는 Shiftbase의 콘텐츠 계층과 컨테이너 설계 원칙을 정의합니다.

## 핵심 관점

Shiftbase의 컨테이너는 강의 영상 묶음이 아니라 실습형·판단형 학습서입니다.

일반 인터넷 글은 다음 흐름을 갖습니다.

```text
읽기 -> 이해는 독자에게 맡김 -> 끝
```

Shiftbase 컨테이너는 다음 흐름을 갖습니다. 다만 모든 페이지가 조작형 인터랙티브일 필요는 없습니다. 핵심은 독자가 더 잘 이해하고 더 잘 판단하는 것입니다.

```text
읽기 -> 시각화/예시 확인 -> 필요할 때 조작 -> 판단 -> 내 상황에 적용 -> 산출물 또는 기준 정리
```

## 콘텐츠 계층

```text
Hub
  Track
    Container
      Module
        Lesson
          Page / Step
            Reading Pane
            Workspace
              Workspace Block
```

## Hub와 Track

허브와 트랙은 탐색과 추천을 위한 내부 분류입니다.

예시:

```text
Hub: Codex

Track: 개발
- Container: Codex로 기존 코드베이스 분석하기
- Container: Codex에게 기능 구현 요청하기

Track: 업무 자동화
- Container: Codex로 리서치 자료 정리하기
- Container: Codex로 CSV 분석 보고서 만들기

Track: 에이전트
- Container: Codex 에이전트 입문
- Container: 다중 에이전트 작업 분해하기
```

사용자가 실제로 선택하는 카드 단위는 컨테이너입니다.

## Container

컨테이너는 다음 기준으로 설계합니다.

- 결과 단위로 정의합니다.
- 대상자와 사용 상황이 구체적이어야 합니다.
- 1개의 명확한 학습 목표를 가져야 합니다.
- 종료 시 학습자가 가져갈 산출물이 있어야 합니다.
- 좌측 서술과 우측 워크스페이스로 나눌 수 있는 페이지 흐름이 있어야 합니다.

나쁜 컨테이너명:

- Codex 활용
- AI 입문
- 업무 자동화
- 프롬프트 엔지니어링

좋은 컨테이너명:

- Codex로 기존 코드베이스 분석하기
- 신입사원을 위한 AI 업무 생존법
- HR 담당자를 위한 AI 온보딩 운영 패키지 만들기
- 재무/회계 담당자를 위한 AI 정산 검토 보조 입문

## Container Card

컨테이너 카드는 최소한 다음 정보를 가져야 합니다.

- 제목
- 대상자
- 난이도
- 예상 소요 시간
- 얻는 산출물
- 무료/유료 여부
- 허브/트랙 태그

예시:

```text
Codex로 기존 코드베이스 분석하기

대상: 기존 프로젝트를 이해해야 하는 개발자/PM
난이도: 입문
시간: 70분
산출물: 코드베이스 구조 요약 + 개선 질문 리스트
태그: Codex, 개발, 분석
```

## Module

모듈은 컨테이너 안에서 학습 흐름을 나누는 장입니다.

좋은 모듈은 하나의 질문에 답해야 합니다.

예시:

- 무엇이 문제인가?
- 내 업무에서 어디에 AI가 들어갈 수 있는가?
- 어떤 결과를 사람이 검토해야 하는가?
- 내 업무에 바로 적용하려면 무엇을 준비해야 하는가?

## Lesson

레슨은 모듈 안의 작은 진행 단위입니다. 하나의 레슨은 여러 페이지를 가질 수 있습니다.

레슨은 영상 강의 하나와 동일한 개념이 아닙니다. 설명, 실습, 시각화, 상태 저장을 포함하는 진행 단위입니다.

## Page / Step

페이지는 사용자가 한 화면에서 경험하는 최소 학습 단위입니다.

페이지는 항상 다음 구조를 기본으로 합니다.

```text
좌측 서술 영역 + 우측 워크스페이스
```

좌측은 설명과 지시를 담당하고, 우측은 보기, 조작, 참고 이미지, 매뉴얼 카드, 비교표, 결과 정리를 담당합니다.

우측 워크스페이스가 항상 복잡한 인터랙티브 도구일 필요는 없습니다. 단순 매뉴얼, 스크린샷, 도식, before/after 표, 예시 문장만으로 더 잘 이해되는 페이지라면 정적 자료를 선택합니다.

## Workspace Blocks

워크스페이스는 여러 블록을 조합해 구성합니다.

초기 표준 블록 후보:

- `image-viewer`: 이미지, 스크린샷, 도식 표시
- `diagram`: 흐름도, 관계도, 단계 구조
- `quiz`: 객관식, OX, 다중 선택
- `card-sort`: 카드를 기준에 맞게 분류
- `block-board`: 블럭 배치
- `checklist`: 체크리스트 선택
- `text-input`: 학습자 입력
- `prompt-builder`: 프롬프트 조립
- `before-after`: 변환 전후 비교
- `table`: 표 또는 비교표
- `result-card`: 산출물 정리
- `state-summary`: 이전 페이지에서 만든 상태 요약

중요한 원칙은 페이지 타입을 늘리는 것이 아니라, 워크스페이스 블록을 표준화하고 조합하는 것입니다.

## 형식 선택 원칙

페이지의 형식은 가장 낮은 복잡도부터 검토합니다.

```text
본문 설명
-> 단순 매뉴얼
-> 참고 이미지/도식
-> 비교표/예시 목록
-> 체크리스트
-> 입력·분류·수정형 인터랙티브
-> code-canvas 기반 도구
```

인터랙티브는 다음 조건을 만족할 때만 사용합니다.

- 사용자가 실제로 선택, 입력, 분류, 수정, 저장 중 하나를 해야 합니다.
- 그 행동이 모듈의 핵심 판단을 강화합니다.
- 정적 매뉴얼이나 참고 이미지보다 이해가 쉬워집니다.
- 결과가 다음 페이지나 최종 산출물에 의미 있게 남습니다.

위 조건을 만족하지 못하면 단순 매뉴얼, 참고 이미지, 비교표, 예시 문장으로 구성합니다.

## 좋은 컨테이너의 리듬

컨테이너는 설명만 이어지면 안 되고, 모든 페이지가 조작만 있어도 피로합니다. 조작은 학습 행동이 필요할 때만 넣습니다.

권장 리듬:

```text
문제 상황 제시
-> 짧은 개념 설명
-> 시각적 예시
-> 필요할 때 조작 실습
-> 결과 확인
-> 내 상황 적용
-> 산출물 정리
```

페이지 단위 리듬:

```text
읽기 -> 보기 -> 해보기 -> 비교하기 -> 적용하기 -> 가져가기
```

## 무료 AI 입문 컨테이너 방향

무료 콘텐츠는 일반적인 "메일 쓰기, 요약하기"에 머물면 약합니다.

직무별 컨테이너로 나누고, 각 컨테이너는 해당 직무의 막막한 순간을 해결해야 합니다.

예시:

- 신입사원: 애매한 지시를 실행계획, 확인 질문, 보고 문장으로 바꾸기
- HR: 온보딩, 채용, 교육, 공지, 설문 운영 패키지 만들기
- 총무/일반사무: 행사, 비품, 업체, 공지, 신청 폼 운영 세트 만들기
- 재무/회계: 정산, 증빙 요청, 비용 분류 후보, 마감 체크리스트 만들기
- 영업관리/영업지원: 고객 요청을 내부 처리 항목과 회신 초안으로 나누기
- 마케팅: 캠페인 아이디어를 브리프, 메시지, 콘텐츠 캘린더로 바꾸기

공통 AI 기본지식은 선택형 컨테이너로 두고, 실제 유입은 직무별 컨테이너가 담당합니다.

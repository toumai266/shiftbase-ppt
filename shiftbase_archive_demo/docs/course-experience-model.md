# Course Experience Model

## ShiftPlayer

Shiftbase의 메인 콘텐츠 런타임은 `ShiftPlayer`다.

ShiftPlayer는 하나의 학습 화면 안에서 두 가지 플레이어를 사용한다.

```text
ShiftPlayer
  SlidePlayer
  ContentPlayer
```

`SlidePlayer`는 기본 개념, 이론, 사례, 관점을 키노트식 발표 무대감으로 이해시키는 사각 슬라이드다.

`ContentPlayer`는 입력, 선택, 비교, 실습, 결과 확인처럼 사용자가 직접 만지는 메인 작업 공간이다.

콘텐츠에 따라 처음부터 실습으로 들어갈 수도 있고, 슬라이드로 개념을 잡은 뒤 실습으로 넘어갈 수도 있다. 중요한 것은 강의 슬라이드와 인터랙티브 실습이 서로 다른 상품처럼 느껴지지 않고 같은 흐름 안에서 자연스럽게 이어지는 것이다.

`ShiftPlayer`, `SlidePlayer`, `ContentPlayer`는 구현과 문서에서 쓰는 구조 이름이다. 사용자 화면에 이 이름을 직접 노출하지 않는다.

## 기본 구조

ShiftPlayer는 화면 성격에 따라 두 가지 구조를 쓴다.

슬라이드 화면:

```text
[풀 폭 슬라이드]
```

슬라이드 화면은 좌우 스플릿을 쓰지 않는다. 개념 설명, 이론, 사례를 하나의 사각 슬라이드로 보여준다.

슬라이드는 문서 본문을 그대로 보여주는 화면이 아니다. 제목, 핵심 문장, 짧은 설명, 작은 참고 블록으로 재구성한다. 길게 읽히는 설명은 실습 화면의 유틸리티 패널이나 메뉴에서 보조로 다룬다.

실습 / 콘텐츠 화면:

학습 페이지의 현재 최소 기준은 13인치 노트북급 데스크톱 화면이다.

```text
minimum viewport: 1280 x 800
```

이보다 작은 화면에서는 학습 화면을 바로 열지 않고 경고 화면을 먼저 보여준다.

현재 범위는 데스크톱 학습 화면이다.

데스크톱 기준 권장 구조:

```text
[유틸리티 패널 30%] [메인 작업 공간 70%]
```

플레이어는 페이지 메뉴바 하단에서 풀브리드로 구성한다. 중앙 카드 안에 갇힌 교재 화면이 아니라, 브라우저 폭을 적극적으로 쓰는 작업 화면이어야 한다.

이전/다음 조작은 별도 푸터 영역을 만들지 않는다. 진행률과 함께 상단 플레이어 바에 작게 배치해 메인 콘텐츠 영역을 최대한 확보한다.

좌측 30%는 설명문을 길게 읽히는 영역이 아니라 유틸리티 공간이다.

역할:

- 실습 세부 설명
- 현재 콘텐츠 메뉴
- 핵심 개념 요약
- 참고 힌트
- AI 챗봇 또는 AI 코치
- 사용자가 막혔을 때의 보조 정보

우측 70%는 메인 작업 공간이다.

역할:

- 슬라이드
- 예시 화면
- 입력창과 선택 카드
- 비교 화면
- 이미지, 표, 자료
- AI 결과 확인
- 사용자가 직접 만지는 실습 블록

## 코스 계층

현재 프로젝트의 콘텐츠 계층은 아래 구조를 유지한다.

```text
Container
  Module
    Lesson
      Page / Step
        Utility Panel
        Workspace
          Workspace Block
```

사용자가 고르는 상품 단위는 `Container`다. 문서에서는 사용자-facing 표현으로 `코스`를 쓸 수 있다.

`Module`은 코스 안의 큰 장이다.

`Lesson`은 URL과 플레이어 진입 단위다.

`Page / Step`은 사용자가 한 번에 보는 최소 화면이다.

## Page / Step 작성 기준

각 Page / Step은 역할이 분명해야 한다.

```text
개념을 보여준다
사례를 보여준다
무엇을 입력한다
무엇을 고른다
무엇을 비교한다
무엇을 판단한다
무엇을 고친다
```

좋지 않은 Step:

```text
AI 프롬프트의 원리를 길게 설명한다.
```

좋은 Step:

```text
짧은 슬라이드로 원리를 보여준 뒤, 두 개의 AI 요청문을 비교하게 한다.
```

좋지 않은 Step:

```text
AI 이미지 생성의 가능성을 소개한다.
```

좋은 Step:

```text
같은 제품 설명으로 만든 이미지 3개를 보고 어떤 것이 더 판매 페이지에 맞는지 판단한다.
```

## 슬라이드와 이론의 위치

이론은 없어지는 것이 아니라 `SlidePlayer`에서 고급스럽고 자연스럽게 다룬다.

권장 처리:

- 슬라이드는 사각형에 가깝게 구성하고 라운딩은 아주 조금만 둔다.
- 슬라이드는 16:9 발표 캔버스에 가깝게 구성한다.
- 슬라이드 바깥은 밝은 회색 무대처럼 처리해 슬라이드 자체가 주인공이 되게 한다.
- 하나의 슬라이드는 하나의 개념, 사례, 관점에 집중한다.
- 슬라이드가 끝나면 같은 흐름 안에서 ContentPlayer 실습으로 이어진다.
- 실습 중 필요한 이론은 좌측 유틸리티 패널이나 AI 코치가 보조한다.

사용자는 강의를 보는 사람인 동시에 직접 만져보는 사람이다. 둘 중 하나만 강요하지 않는다.

## Workspace Block

현재 구현된 블록 타입은 다음이다.

```text
code-canvas
intro-summary
image-definition
block-board
state-summary
example-list
candidate-select
quiz
checklist
result-card
```

블록 선택 기준은 단순하다.

- 보여주는 것이 목적이면 `image-definition`, `example-list`, `code-canvas`
- 사용자가 고르는 것이 목적이면 `quiz`, `checklist`, `candidate-select`
- 사용자가 상태를 만들고 이어가는 것이 목적이면 `block-board`, `state-summary`
- 결과를 정리하는 것이 목적이면 `result-card`

복잡한 `code-canvas`는 실제로 만져야 할 화면이 있을 때만 쓴다.

## AI 코치의 위치

챗봇은 별도 상담창이 아니라 현재 Step에 붙어 있는 보조 코치여야 한다.

좋은 답변:

```text
지금 화면의 두 번째 선택지는 근거가 약합니다. 출처가 없고 숫자의 기준 시점도 없기 때문입니다.
```

나쁜 답변:

```text
AI 활용에서는 명확한 맥락과 목표가 중요합니다.
```

AI 코치는 현재 단계, 사용자의 선택, 화면의 자료를 알고 답해야 한다.

## 완료 기준

코스 완료는 영상을 다 봤는지가 아니라 실습 흐름을 수행했는지로 판단한다.

초기 완료 조건:

- 핵심 Step을 방문했다.
- 필수 입력 또는 선택을 했다.
- 마지막 비교/판단 화면까지 도달했다.

상용 단계 완료 조건:

- 필수 블록의 `writes` 상태가 저장됐다.
- 최종 `result-card`가 비어 있지 않다.
- 코스 버전에 맞는 진행 상태가 서버에 저장됐다.
